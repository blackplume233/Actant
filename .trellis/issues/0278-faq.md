# FAQ 对齐治理 Issue 草案

## 标题
对齐 FAQ 产品定位，开展仓库历史内容审查、概念收缩与现状整理

## 背景

基于近期重新回看 AgentCraft FAQ（Actant 前身）的产品表述，需要对当前仓库进行一次更系统的“历史内容审查 + 概念治理 + 现状整理”。

这次工作的重点不是立刻改某一块代码，而是先回答一个更基础的问题：

**当前仓库中的文档、Spec、设计稿、实现说明、规划文件、验证资产，是否仍然共同服务于同一个产品模型？**

从 FAQ 反推，当前应该统一并收敛出的产品基线至少包括：

- Actant 应被稳定表达为一个**底层平台**
- 它的职责是支撑上层 **App / SOP / CI / 引擎集成** 等交付形态
- Agent Instance 的递进模型应是 `repo -> service -> employee`
- 其中 `service` 很可能是当前最常见、最重要的主交付形态
- `employee` 应被视为建立在 `service` 之上的自治增强形态，而不是默认主体
- 当前很多能力（Plugin / Scheduler / Memory / Email / Initializer / Context Injection）应被放回“平台底座能力”的语境中理解，而不是继续被拆成松散 feature backlog

目前仓库已经有一部分内容在朝这个方向收敛，但也仍然存在明显的历史包袱：

- 不同时期形成的文档叙事彼此并不完全一致
- 某些旧设计稿仍在隐性充当“概念基线”
- 一些术语已经发生语义漂移，但仓库里尚未完成收口
- 一些能力在实现层已有基础，但在规划层仍被当作“待从零设计”
- Spec 和 endurance asset 还没有真正围绕 archetype 和平台层级组织起来

因此，本 issue 的目的不是补一篇文档，而是为后续一轮**概念收缩、历史清理、文档收敛、规划对齐、验证重组**建立统一入口。

## 问题陈述

当前仓库的主要问题，不是“缺某一个功能说明”，而是**知识层、规划层、实现层、验证层之间的叙事和边界没有完全对齐**。

更具体地说，当前存在以下几类风险：

- **历史叙事残留**
  - 仓库中保留了不同阶段的产品理解和设计表达，部分内容仍代表旧阶段的判断
- **概念外延膨胀**
  - 一些术语在不同文件中被赋予过多含义，导致同名异义
- **概念层级混叠**
  - template/domain-context 层与 platform/runtime-services 层经常被混用
- **主次关系失焦**
  - `employee` 作为“增强形态”在部分文本中被抬升成默认主体，稀释了 `service` 的主交付角色
- **规划与实现脱节**
  - 代码里已有的基础能力，没有被规划和文档正确承认与复用
- **验证视角仍偏 backlog**
  - 验证资产更像 issue/phase 的附属物，而不是产品模型的验收体系

这意味着：即使后续继续推进功能开发，如果不先做一轮概念治理，仓库仍会继续累积“局部正确、整体漂移”的问题。

## 本次 issue 的核心目标

本 issue 旨在推动以下五件事：

### 1. 做一次完整的历史内容审查
不只看“当前写法对不对”，还要审查：

- 哪些文档属于旧阶段表述
- 哪些历史设计稿仍在影响当前理解
- 哪些规划文件已经与当前基线不一致
- 哪些内容应该保留为历史记录，哪些应升格为当前基线，哪些应明确标记为过时

### 2. 做一次概念收缩
重点不是继续发明新术语，而是把现有概念压缩、去重、边界化，避免无限增殖。

特别需要收缩的包括：

- Actant 的产品定位
- `repo / service / employee`
- Plugin 的双重语义
- Workflow / MCP / Prompt / Skill 与平台能力的分界
- Domain Context 与 platform services 的层级关系

### 3. 做一次概念整理
在收缩之后，要把当前仍然成立的概念重新组织成一致结构，让后续文档、Spec、实现、测试都能引用同一套框架。

### 4. 对仓库现状做一次结构化整理
把“当前仓库实际上已经有什么”讲清楚，而不是继续让 planning、design、wiki、code 各说各话。

包括但不限于：

- 当前已经稳定存在的概念模型
- 当前代码里已经具备的 runtime substrate
- 当前 dashboard 已经体现出的产品模型
- 当前 spec 已经表达出的上层/下层结构
- 当前验证资产已经覆盖与尚未覆盖的边界

### 5. 形成一份后续整改路线
最终目标不是只输出“扫描结论”，而是形成一份可拆解、可追踪、可验收的整改清单，用于指导后续文档、Spec、规划、实现与验证同步收敛。

## 本次工作的判断基线

### 产品定位基线
- Actant 是底层平台，而不是单一 Agent App
- Actant 不应只被描述成“Agent 管理器”或“Agent 工具箱”
- Actant 的价值在于为上层 App / SOP / CI / 引擎集成提供统一底座

### Agent 形态基线
- `repo`：工作区 / 资产载体 / 开发入口形态
- `service`：默认主交付形态，被动响应、生命周期受平台管理
- `employee`：建立在 `service` 之上的自治增强形态，包含调度、心跳、持续监控等能力

### 分层基线
- template / domain-context 层：
  - Skill
  - Prompt
  - Workflow
  - MCP
  - SubAgent
  - agent-side plugin（如保留）
- platform / runtime-services 层：
  - PluginHost / system plugin
  - Context Injection
  - Scheduler
  - Memory
  - Email
  - Budget / KeepAlive
  - EventBus / Hook runtime
  - Initializer substrate

### 验证基线
- Spec 与 endurance 需要能证明：
  - `repo` 是否满足其承诺
  - `service` 是否满足其承诺
  - `employee` 是否满足其承诺
- 验证体系应围绕产品模型与平台层次构建，而不是只围绕 issue 清单

## 对过去内容的审查要求

本次 issue 不应只做“最新文档修正”，还要对仓库中的过去内容做系统审查，并分类处理。

### 需要被识别的历史内容类型

#### 1. 仍然有效、可作为当前基线保留的内容
这类内容应被明确标记为“当前有效”或被引用进核心文档。

#### 2. 部分有效，但叙事已过时的内容
这类内容可以保留事实记录，但不应继续作为当前设计依据。

#### 3. 已与当前产品基线冲突的内容
这类内容需要：
- 重新改写
- 降级为 historical
- 或在文档顶部显式标注 superseded / archived context

#### 4. 虽然过时，但具有演进历史价值的内容
这类内容不应直接删除，但应防止其继续参与当前概念定义。

### 过去内容审查的目标
- 识别“哪些旧内容还在隐性定义今天的产品”
- 识别“哪些旧术语还在绑架今天的讨论”
- 识别“哪些旧规划已不适合继续作为 Phase 4 的叙事入口”

## 概念收缩目标

本次 issue 需要明确推动一轮“概念收缩”，防止仓库继续在术语和层级上发散。

### A. 收缩产品主叙事
避免以下几种并行而未收口的表达长期共存：

- “Docker for AI Agents”
- “Agent 管理器”
- “Agent 平台”
- “Agent App 容器”
- “企业级 Agent runtime”
- “自治 Agent 系统”

这些说法不是完全错误，但需要收敛成有主次的统一叙事。

建议目标：
- 主叙事：Actant 是底层平台
- 辅助比喻：可保留 Docker 类比，但不能代替平台定位
- 场景表达：上层 App / SOP / CI / 引擎集成

### B. 收缩 Agent archetype 叙事
避免 `repo / service / employee` 成为“只是三个并列名词”。

建议目标：
- 它们不是三个平级 marketing label，而是**管理深度递进模型**
- `service` 不是中间态配角，而是默认主交付形态
- `employee` 不应继续被当作默认主角

### C. 收缩 Plugin 概念
当前最需要收缩的概念之一是 `plugin`。

至少应区分：

- agent-side plugin
  - 作用于 workspace / backend / agent 自身
  - 属于模板物化或 agent 侧扩展
- actant-side plugin / system plugin
  - 作用于平台 runtime
  - 由 PluginHost 承载
  - 服务于 scheduler / memory / email / hook / initialization 等平台能力

如果不做这一步，#14 将持续处于高歧义状态。

### D. 收缩 Domain Context 概念
当前需要避免把所有能力都归到 Domain Context 名下。

建议目标：
- Domain Context 只描述 agent 作为实例被赋予的上下文拼装
- 不再把平台底座能力模糊地等同于 Domain Context 组件
- 明确哪些内容是“随模板引用/物化”，哪些是“由平台 runtime 注入/托管”

### E. 收缩验证概念
当前“测试很多”不等于“产品模型被验证”。

建议目标：
- 验证从 issue 驱动收缩到 archetype 驱动
- 验证从 feature checklist 收缩到 platform contract

## 概念整理目标

在完成概念收缩之后，需要把仍然成立的概念重新组织成一套清晰结构。

建议整理成以下几层：

### 1. 产品层
回答：
- Actant 是什么
- 它与上层 App / SOP / CI / engine integration 的关系是什么

### 2. Archetype 层
回答：
- 为什么是 `repo -> service -> employee`
- 各自面向什么交付形态
- 各自的默认能力边界是什么

### 3. Template / Domain Context 层
回答：
- Skill / Prompt / Workflow / MCP / SubAgent / agent-side plugin 的职责是什么
- 它们如何被引用、解析、物化

### 4. Platform / Runtime Services 层
回答：
- Context Injection、PluginHost、Scheduler、Memory、Email、Budget、Hook Runtime、Initializer 等能力如何协同

### 5. 验证层
回答：
- 这些层次如何被 Spec 和 endurance 覆盖
- 如何从产品目标映射到验证资产

## 仓库现状整理目标

本次 issue 还需要对“仓库现在到底处于什么状态”做一次系统整理。

### 一、整理当前已经成立的能力底座
需要明确承认并记录：

- `AgentManager` 已经不是简单生命周期壳子
- `SessionContextInjector` 已经是可扩展基础设施
- `Initializer Pipeline` 已经存在 StepRegistry / rollback / timeout / shared state 基础
- dashboard 已经比部分文档更先进入 archetype-first 模型
- config/spec 已经开始具备 App / archetype / schedule 等结构化表达

### 二、整理当前仍然混乱的区域
需要明确记录：

- 架构叙事仍有旧版残留
- Plugin 术语仍然双义
- 历史 design doc 与当前 planning 的状态表达存在冲突
- 验证资产尚未围绕 archetype 重组

### 三、整理当前规划与实现之间的断裂点
例如：

- #14 不应再被理解为当前 `PluginManager` 的继续扩展
- #37 不应再被理解为纯从零设计
- scheduler/memory/email 不应继续作为松散 backlog feature，而应回到平台底座能力栈中讨论

## 现有冲突 / 漂移 / 需要收敛的重点问题

### 1. 产品叙事仍不够集中
- 一部分文档仍以旧比喻或功能平铺为主
- “底层平台”定位还没有在关键文档中形成单一主叙事

### 2. `service` 的主形态地位不够突出
- 概念上已经存在，但在 roadmap、recipe、design、feature narrative 中尚未被稳定提升为默认主交付形态

### 3. `employee` 仍有被过度中心化的倾向
- 部分规划和示例仍容易让人误解为“未来主体就是 employee”
- 这会误导 roadmap 节奏、实现重心和验证策略

### 4. Plugin 概念继续混用
- 当前代码里的 plugin manager 与 Phase 4 的 Plugin Core 不是一回事
- 若不拆名，后续仍会持续混淆

### 5. 分层表达不统一
- Workflow / MCP / Plugin / Scheduler / Memory / Email 的层级关系在不同文件中不一致

### 6. 历史设计稿仍在隐性影响当前讨论
- 某些 design 文档虽然已经部分过时，但没有被显式降级
- 它们仍可能被团队成员视作当前依据

### 7. 验证体系还没有围绕 archetype 建立主验收逻辑
- 当前更像“功能做完了就补测试”
- 还不是“平台模型成立了，所以这些测试存在”

## 优先级建议

### P0
- 持续误导产品定位的叙事冲突
- 会导致 Phase 4 主线跑偏的 archetype 主次冲突
- system plugin 与 agent-side plugin 的命名和边界冲突

### P1
- platform/runtime-services 与 template/domain-context 的分层不清
- planning 与实现现状表达不一致
- 验证体系尚未围绕 archetype 组织

### P2
- 旧 design 文档老化
- recipe 与展示层叙事不平衡
- 措辞、命名、示例分布不统一

## 验收标准

### 内容审查层
- 已完成对关键历史文档、规划文档、架构文档、Spec 的审查
- 每类历史内容都已被归类为 current / partial / historical / superseded 之一

### 概念治理层
- 已形成一套收缩后的产品叙事
- 已形成一套明确的层级化概念结构
- `service` 与 `employee` 的主次关系已不再模糊
- plugin 双重语义已被拆分或明确标注边界

### 仓库整理层
- 已形成一份“当前仓库现状整理”
- 能清楚说明哪些能力已存在，哪些仍缺失，哪些只是文档段落

### 验证层
- 已为后续 archetype 验收矩阵和 platform-level endurance 重组建立明确入口
- 验证资产不再只按 issue backlog 组织，而开始围绕产品模型收敛
